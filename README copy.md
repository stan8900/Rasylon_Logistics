# Telegram бот для рассылок логистической компании

Telegram-бот помогает запускать и управлять авторассылкой сообщений по группам, где бот состоит и может отправлять сообщения (админские права не обязательны).

## Возможности

- `/start` — приветствие с быстрым меню.
- `/help` — контакт поддержки: `@rasylon_support`, наш ИИ-агент.
- **Для клиентов**: «💰 Пополнить баланс» и «📜 История оплат».
- **Для администраторов**: «⚒ Авторассылка» (кнопка видна всем пользователям, чтобы каждый мог запустить общую рассылку), «📊 Статистика» с периодами день/неделя/месяц/всё, «📋 Выбрать группы», «⚙️ Настройки», «📜 Оплаты», «🔁 Перепроверить оплату» (ручное подтверждение/отклонение, если платёж не появился автоматически), «📄 PDF оплат» (выгрузка отчёта), а также новые инструменты «📥 Парсер» и «📨 Инвайтер».
- Авторассылка работает во всех группах, где бот остаётся участником и может писать. Перед отправкой бот проверяет доступность чатов и сообщает об ошибках.
- Каждый пользователь настраивает свою авторассылку — текст, интервал и выбранные группы сохраняются индивидуально и не влияют на чужие настройки.
- **Парсер аудитории** собирает логины из комментариев последних N постов выбранного канала, сохраняет результат в `data/dumps` и сразу присылает `.txt`.
- **Парсер групп** отображает список чатов, где подключён личный аккаунт, и выгружает участников выбранной группы (через Telethon-подключение, поддерживает прокси, указанные в сессии).
- **Invite Engine** принимает файл с логинами и orchestrates приглашение через несколько личных аккаунтов Telegram с заданными лимитами, задержками и количеством потоков (команда `/jobs` показывает активные/прошлые задачи).
- Для каждого подключённого личного аккаунта можно задать SOCKS4/5 или HTTP-прокси (кнопка «🌐 Прокси» в меню «📱 Номер»).

## Подготовка окружения

1. Установите зависимости:

   ```bash
   python -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   ```

2. Создайте `.env` в корне проекта и пропишите токен бота и параметры доступа:

```env
BOT_TOKEN=ваш_telegram_token
# для продакшена: PostgreSQL на Railway, например
DATABASE_URL=postgresql://user:password@host:5432/dbname
DATABASE_URL_REQUIRED=false
# необязательный список Telegram ID, которым всегда приходят уведомления об оплатах
ADMIN_IDS=12345678,98765432
ADMIN_CODE=TW13
# путь до SQLite-файла с данными (используется, если DATABASE_URL не задан)
   STORAGE_PATH=data/storage.db
   # опционально: данные персонального аккаунта, если рассылка должна идти от лица человека
   TG_USER_API_ID=123456
   TG_USER_API_HASH=abcdef0123456789abcdef0123456789
   TG_USER_SESSION=1AAgbC... (строка сессии из Telethon)
   # необязательно: прокси для общего TG_USER_SESSION, формат host:port[:login[:password]] или schema://login:pass@host:port
   TG_USER_PROXY=socks5://login:pass@127.0.0.1:9050
   # необязательно: режим спячки для личных сообщений боту
   BOT_SLEEP_FROM=00:00
   BOT_SLEEP_TO=09:00
   BOT_SLEEP_TIMEZONE=Asia/Tashkent
   # разовая спячка до конкретной даты/времени, если нужно
   BOT_SLEEP_UNTIL=2026-05-01 09:00
   BOT_SLEEP_MESSAGE=Бот находится в режиме спячки до {until}. Напишите позже.
   ```

Если указать `DATABASE_URL`, бот подключится к PostgreSQL (например, Railway). При пустом значении переменной используется локальный SQLite-файл по пути `STORAGE_PATH` (по умолчанию `data/storage.db`). Переменная `DATABASE_URL_REQUIRED=true` отключает автоматический fallback на SQLite — пригодится на проде, где отсутствие БД должно приводить к ошибке.

По умолчанию ночью с `00:00` до `09:00` по времени Узбекистана (`Asia/Tashkent`) бот на любое личное сообщение отвечает, что находится в режиме спячки до времени включения. Интервал можно поменять через `BOT_SLEEP_FROM` и `BOT_SLEEP_TO`; чтобы отключить ночной офф, задайте им одинаковое время, например `BOT_SLEEP_FROM=00:00` и `BOT_SLEEP_TO=00:00`. Для разовой спячки до конкретного момента используйте `BOT_SLEEP_UNTIL`; поддерживаются форматы `HH:MM`, `YYYY-MM-DD HH:MM`, `YYYY-MM-DDTHH:MM` и `DD.MM.YYYY HH:MM`. Текст можно переопределить через `BOT_SLEEP_MESSAGE`, оставив `{until}` там, где нужно подставить время окончания.

   > При первом запуске данные мигрируют в SQLite. Если рядом лежит старый `storage.json`, он будет автоматически импортирован и больше не используется.

   Если не указывать параметры `TG_USER_*`, сообщения будут отправляться стандартным бот-аккаунтом.

### Рассылка от личного аккаунта

Чтобы сообщения приходили «от лица человека», подключите любую активную учётную запись Telegram:

1. Получите `API ID` и `API Hash` на [my.telegram.org](https://my.telegram.org). Они нужны только вам и не видны пользователям.
2. Сгенерируйте строку сессии Telethon (вам понадобится номер телефона и SMS/код от Telegram):

   ```bash
   python - <<'PY'
   from telethon.sync import TelegramClient
   from telethon.sessions import StringSession

   api_id = int(input('API_ID: '))
   api_hash = input('API_HASH: ')
   with TelegramClient(StringSession(), api_id, api_hash) as client:
       print('TG_USER_SESSION=', client.session.save(), sep='')
   PY
   ```

   Скрипт один раз спросит номер телефона и код подтверждения. Полученную строку скопируйте в `.env` как `TG_USER_SESSION`.
3. Следите, чтобы этот аккаунт состоял во всех группах/каналах, куда нужно отправлять сообщения. Бот по‑прежнему используется только для управления через меню и может не состоять в этих чатах.

### Личные номера пользователей

Теперь каждый оператор может авторизовать личный аккаунт прямо в боте и рассылать сообщения от своего имени. Для этого достаточно оставить в `.env` `TG_USER_API_ID` и `TG_USER_API_HASH` (общая `TG_USER_SESSION` остаётся опциональной и используется, пока пользователь не подключит свой номер).

- В разделе «⚒ Авторассылка» появится кнопка «📱 Номер»: оттуда можно выбрать активный аккаунт, обновить список чатов или нажать «➕ Добавить номер».
- При добавлении бот попросит указать телефон, подтвердить код и, если нужно, пароль двухфакторной авторизации. После успешной авторизации чаты номера автоматически подтянутся в бота.
- Каждый пользователь видит только свои номера и свои группы — чужие чаты недоступны.
- Если личный аккаунт выбран, рассылка идёт только по его чатам и не пересекается с другими операторами.

При успешной настройке авторассылка запускает Telethon-клиент и отправляет сообщения именно от указанного пользователя. Если параметры не заданы или сессия станет невалидной, система вернётся к стандартной отправке от бота.

### Прокси для общей пользовательской сессии

Если общий `TG_USER_SESSION` должен работать через купленные прокси (например, чтобы Telegram не видел ваш реальный IP), пропишите их в `.env`:

1. Добавьте `TG_USER_PROXY=socks5://login:pass@host:port` (поддерживаются схемы `socks5`, `socks4` и `http`, также можно использовать короткий формат `host:port[:login[:password]]`).
2. Перезапустите бота. Все действия, которые используют общий пользовательский клиент (авторассылка без личных номеров, парсер, инвайтер), автоматически пойдут через этот прокси.
3. Чтобы отключить прокси, удалите переменную или оставьте её пустой и перезапустите сервис.

Если править `.env` неудобно, администраторы могут задать прокси прямо в боте: в главном меню нажмите «🌐 Общий прокси», отправьте строку подключения в том же формате или напишите `off`, чтобы вернуться к прямому соединению. Настройки сохраняются в базе и вступают в силу сразу — общая сессия переподключится с новыми параметрами.

### Прокси для личных аккаунтов

Чтобы разграничить трафик и разгружать IP-адреса, можно привязать индивидуальные прокси к каждому подключённому номеру:

- В меню «📱 Номер» нажмите кнопку «🌐 Прокси» рядом с нужным аккаунтом.
- Бот примет строку вида `socks5://login:pass@127.0.0.1:9050` или короткий формат `host:port[:login[:password]]`. Поддерживаются схемы `socks5`, `socks4` и `http`.
- Команда `off` (или `none`) отключает прокси и возвращает прямое подключение.
- Выбранные параметры сохраняются в хранилище и автоматически применяются как для авторассылки, так и для парсера/инвайта.

### Парсер аудитории

Кнопка «📥 Парсер» в главном меню доступна администраторам. Процесс:

1. Введите @username или ссылку на канал.
2. Укажите, сколько последних постов анализировать (1–500).
3. Бот подключится через MTProto (используется общий `TG_USER_SESSION` или выбранный личный аккаунт), выгрузит всех авторов комментариев и сохранит `.txt` в `data/dumps`.
4. Готовый файл сразу приходит в чат; список последних выгрузок можно посмотреть командой `/dumps`.

Формат файла: `user_id,username,access_hash` — подходит для дальнейшей обработки или загрузки в CRM.

### Парсер групп

Кнопка «👥 Парсер групп» показывает доступные чаты (группы/супергруппы), в которых состоит личный аккаунт, привязанный к MTProto. Процесс:

1. Выберите группу из списка или отправьте её @username/ID.
2. Бот подключится тем же пользовательским клиентом (через его прокси/сессию) и выгрузит всех участников через `iter_participants`.
3. Результат сохраняется в `data/dumps` и отправляется в чат (тот же CSV-формат `user_id,username,access_hash`).

Этот режим полезен, когда нужно собрать аудиторию из закрытых групп, где работает личный аккаунт оператора; сам бот может даже не состоять в этих чатах.

### Invite Engine (многопоточный инвайтер)

Кнопка «📨 Инвайтер» запускает мастер настройки массовых приглашений:

1. Загрузите `.txt` с логинами (по одному `@username` в строке). Файл сохраняется в `data/uploads`.
2. Укажите чат/канал, куда нужно приглашать.
3. Задайте лимиты: сколько приглашений сделает один аккаунт, задержку (секунды) и число параллельных потоков. Система автоматически добавляет небольшую рандомизацию задержек, чтобы не ловить флуд-баны.

Для отправки инвайтов нужны авторизованные личные аккаунты через меню «📱 Номер». Каждый аккаунт работает в собственной Telethon-сессии и соблюдает ограничения:

- лимит приглашений на аккаунт за задачу;
- задержка между инвайтами;
- общее число одновременно работающих аккаунтов (thread pool).

Прогресс по заданиям отслеживается командой `/jobs` — показываются последние 10 задач с количеством успешных приглашений и статусом (`pending`, `running`, `completed`, `failed`). После завершения задания файл с логинами остаётся в `data/uploads`, так что можно перезапустить задачу с другими лимитами.

3. Запустите бота:

   ```bash
   python bot.py
   ```

### Веб-админка

Отдельная админ-панель доступна как сайт в папке `web_admin/` и читает статистику напрямую из той же базы данных, что и бот. Для неё есть отдельный набор зависимостей:

```bash
pip install -r web_admin/requirements.txt
ADMIN_WEB_PASSWORD=надежный_пароль python -m web_admin.main
```

По умолчанию сайт запускается на `http://0.0.0.0:8080`. Порт можно поменять через `ADMIN_WEB_PORT` или `PORT`. Если `ADMIN_WEB_PASSWORD` не задан, используется `ADMIN_CODE`. Старый запуск `python admin_web.py` тоже оставлен и просто открывает новый модуль.

В веб-панели есть переключение периодов день/неделя/месяц/всё, график динамики, карточки оплат/подписок/рассылок и таблица последних оплат.

## Как пользоваться

1. Запустите диалог командой `/start`.
2. Клиенты выбирают «💰 Пополнить баланс», переводят средства на карту 9860 1701 1433 3116 и вводят номер своей карты и имя. Статус можно проверить через «📜 История оплат».
3. Администраторы выполняют `/admin` (или `/админ`), вводят код `TW13` и получают доступ к панели.
4. Добавьте бота в рабочие группы и убедитесь, что он может отправлять сообщения (админские права — по желанию).
5. В разделе «⚒ Авторассылка» (он открыт и для администраторов, и для обычных пользователей) задайте текст, интервал (в минутах) и отметьте группы — эти параметры сохраняются отдельно для каждого пользователя.
6. Нажмите «▶️ Запустить», чтобы включить рассылку. Бот отправит сообщение во все выбранные чаты и будет повторять отправку по расписанию; запускать авторассылку могут все пользователи, но только после подтверждённой оплаты. Если платеж прошёл, но не отобразился, администратор использует «🔁 Перепроверить оплату» и вручную подтверждает/отклоняет заявку.
7. В разделе «📜 Оплаты» администратор видит список пополнений: имя, username, карту и срок действия подписки.
8. В любой момент можно остановить рассылку, изменить текст, интервал или список групп.

## Структура проекта

- `bot.py` — точка входа, регистрация хэндлеров и запуск бота.
- `web_admin/` — отдельная веб-админка со своим `requirements.txt`; статистику берёт напрямую из PostgreSQL/SQLite базы.
- `app/auto_sender.py` — фоновая отправка сообщений по расписанию, проверка оплаты и обработка ошибок доставки.
- `app/storage.py` — хранилище настроек, известных чатов, оплат и сессий (работает с PostgreSQL или SQLite).
- `app/keyboards.py` — inline-клавиатуры для управления ботом.
- `app/states.py` — состояния FSM для диалогов.
- `data/storage.db` — локальная SQLite-база (используется, если не настроен `DATABASE_URL`; создаётся автоматически, старый JSON мигрирует в неё при запуске).

## Полезно знать

- Используйте `/cancel` для отмены текущего шага настройки.
- Если бот покидает чат или получает ограничения на отправку сообщений, группа автоматически удаляется из списка рассылки.
- Статистика показывает заявки на оплату, оформленные и активные подписки, расчёт выручки по подтверждённым оплатам, количество запусков рассылок, успешных отправок и активных рассылок за выбранный период.
- Перечень администраторов задаётся либо командой `/админ`, либо в `ADMIN_IDS`; только они получают запросы на подтверждение оплаты.
- Авторассылка активируется только после успешного пополнения баланса (подтверждённая админом заявка).
- Администраторы могут выгрузить текущий список оплат в PDF через «📄 PDF оплат» — файл генерируется на лету и отправляется в чат.

## Деплой на Railway

Для Railway лучше использовать два сервиса из одного GitHub-репозитория:

- `sendertistics-web` — веб-админка, start command: `python admin_web.py`
- `sendertistics-bot` — Telegram-бот, start command: `python bot.py`

Оба сервиса должны быть подключены к одной PostgreSQL-базе Railway через общий `DATABASE_URL`. Локальный SQLite на Railway использовать не стоит: файловая система деплоя не предназначена для постоянного хранения рабочей базы.

### 1. Подготовьте проект

1. Загрузите репозиторий на GitHub.
2. В Railway создайте новый Project.
3. Добавьте PostgreSQL Database в этот Project.
4. Добавьте первый сервис из GitHub repo для веб-админки.
5. Добавьте второй сервис из того же GitHub repo для бота.

Railway/Nixpacks установит зависимости из `requirements.txt`; текущий `nixpacks.toml` уже оставлен для системных библиотек, нужных Python-зависимостям.

### 2. Настройте веб-сервис

В сервисе веб-админки задайте Custom Start Command:

```bash
python -m web_admin.main
```

Переменные окружения для веб-сервиса:

```env
DATABASE_URL=${{Postgres.DATABASE_URL}}
ADMIN_WEB_PASSWORD=надежный_пароль_для_входа
ADMIN_CODE=TW13
PAYMENT_AMOUNT=100000
PAYMENT_CURRENCY=UZS
PAYMENT_VALID_DAYS=30
```

Если платформа позволяет выбрать файл зависимостей для сервиса, укажите `web_admin/requirements.txt`. После деплоя откройте публичный домен сервиса. Вход выполняется по `ADMIN_WEB_PASSWORD`; если он не задан, используется `ADMIN_CODE`.

### 3. Настройте bot-сервис

В сервисе Telegram-бота задайте Custom Start Command:

```bash
python bot.py
```

Переменные окружения для bot-сервиса:

```env
DATABASE_URL=${{Postgres.DATABASE_URL}}
DATABASE_URL_REQUIRED=true
BOT_TOKEN=ваш_telegram_token
ADMIN_IDS=12345678,98765432
ADMIN_CODE=TW13
PAYMENT_AMOUNT=100000
PAYMENT_CURRENCY=UZS
PAYMENT_VALID_DAYS=30
TG_USER_API_ID=123456
TG_USER_API_HASH=abcdef0123456789abcdef0123456789
TG_USER_SESSION=1AAgbC...
TG_USER_PROXY=
BOT_SLEEP_FROM=00:00
BOT_SLEEP_TO=09:00
BOT_SLEEP_TIMEZONE=Asia/Tashkent
```

`TG_USER_*` нужны только если рассылка должна идти через личный аккаунт Telegram. Если бот отправляет сам, их можно не задавать.

### 4. Проверка после деплоя

- В веб-сервисе откройте `/health`; должен вернуться `{"ok": true}`.
- Откройте домен веб-сервиса и войдите в админку.
- В логах bot-сервиса должна появиться строка запуска бота.
- В Telegram выполните `/start`, затем проверьте оплату/рассылку и обновление статистики в веб-панели.

Если Railway пишет `No start command could be found`, значит Custom Start Command не задан в конкретном сервисе.

## Деплой на Pella

1. Перейдите в корневую папку проекта (`Bot`).
2. Создайте архив без вложенной папки, чтобы `bot.py` лежал на верхнем уровне:

   ```bash
   zip -r bot.zip .
   ```

3. Загрузите `bot.zip` в Pella и укажите команду запуска `python /app/bot.py`.

> Ошибка `python: can't open file '/app/bot.py'` появляется, если архив содержит дополнительную вложенную директорию и `bot.py` оказывается по пути `/app/что-то/bot.py`. Убедитесь, что архив сформирован из содержимого каталога, а не из родительской папки.
